## 1. Do these first (top 5)

1. **Close the two cross-tenant IDOR holes.**
   `internal/handler/session.go:186-431` (Unassign, RemoveStripTarget, ToggleChecklistItem hit unscoped `WHERE id=$1` queries at `repository/session.go:155,361,688`) and `internal/handler/training.go:87-195` (Get/Update/AddItem/UpdateItem never compare `plan.LocationID` to the URL).
   Why now: any authenticated setter at any gym can mutate other orgs' sessions and training plans by UUID. With one gym today the blast radius is zero — the day gym #2 signs up it's a live vulnerability. The fix pattern already exists in the same files (resolveSession, card_batch.go's `LocationID != locationID` checks); scope repo writes by `session_id` and reject on `RowsAffected==0`. Effort M.

2. **Stand up real Postgres backups + a restore runbook.**
   The only `pg_dump` in the repo is the dev-refresh at `Makefile:93-98`; the prod safety net is Fly's ~5-day same-platform volume snapshots. Nightly `pg_dump -Fc` (GHA cron or Fly machine) to a separate Tigris/S3 bucket, 30+ day retention, documented `pg_restore` commands, one restore drill against staging.
   Why now: the pilot gym's data is already irreplaceable, and a bad migration backfill or fat-fingered DELETE noticed after day 5 currently has no recovery path. Effort M.

3. **Make session Publish atomic and retry-safe.**
   `internal/handler/session.go:264-317` runs strip-archive → PublishSessionRoutes → UpdateStatus as separate pool calls. A failure (or plain double-submit — there's no status guard) re-runs the whole-wall strip and archives the routes the previous attempt just published, with no way to restore them.
   Why now: publish is the pilot's core recurring workflow and the failure mode destroys the gym's live route board. Minimal fix: one repo method wrapping all steps in `database.RunInTx` (`internal/database/tx.go:13`) + a session-status guard. The full querier-interface refactor can come later (see §3). Effort M.

4. **Stop self-inflicted outages (three S-effort ops fixes, one bundle).**
   - `/health` returns 503 on a live S3 HeadBucket failure (`internal/handler/health.go:64`, `service/storage.go:170`); with `min_machines_running=1`, a transient Tigris blip pulls the only machine from routing — total outage of a working app. Make the Fly probe process+DB only; report storage in the body.
   - Graceful shutdown never waits: `cmd/api/main.go:150` falls through while `Shutdown()` is still draining, closing the pgx pool under in-flight requests and dropping queued async events (badges/notifications) on **every deploy**. Add a done-channel wait.
   - No `kill_timeout` in `fly.toml` — Fly's default 5s SIGKILL undercuts the 15s drain budget (main.go:138). Set `kill_timeout = "20s"`.
   Why now: merges auto-deploy to prod, so the shutdown race fires on every merge, and the health-check coupling can down the app any day.

5. **Log every 500 and fix the latent nil-deref.**
   `handler.Error()` (`internal/handler/helpers.go:24`) writes JSON and discards the cause; 219 InternalServerError sites, ~half the API (session, auth, wall, location, quest, ascent, analytics, training, follow, card) has zero slog calls. Plus `route.go:151` does `rt, _ = h.routes.GetByID(...)` then dereferences — a transient DB error panics and loses the audit record; `route.go:291` drops SetTags errors and returns 200.
   Why now: you are the only support engineer for the pilot. A reported 500 from half the API is currently undiagnosable. Add one `InternalError(w, r, msg, err)` helper that slogs with request ID, sweep the sites, and check the two route.go reloads. Effort M.

## 4. Security-relevant items (explicit)

Confirmed, in priority order:
1. **Cross-tenant IDOR** — session assignments/strip-targets/checklist (handler/session.go:186-431) and training plans (handler/training.go:87-195). Authenticated cross-org read/write by UUID. Launch blocker for tenant #2.
2. **Spoofable client IP via chi RealIP** (router.go:42) — defeats the login brute-force limiter (fresh bucket per forged header) and exposes pool diagnostics through the `"172."` prefix check in health.go:24. High.
3. **No rate limiting on the authenticated API** (router.go:434), including CPU-heavy per-route card rendering on a 1GB VM — single-client DoS. Med.
4. **Audit-log blind spots** — location/settings/session mutations and view-as impersonation leave no trail while lower-impact tag edits do (settings.go, auth.go:309). Med, forensics gap.
5. **clientIP IPv6 truncation** (security.go:376) — distinct IPv6 clients share rate-limit buckets, weakening the auth throttle and causing spurious 429s. Med.
6. **View-as privilege-downgrade path untested** with confirmed comment/code drift in SetViewAs (websession.go:314, auth.go:328) — downgrade-only today, but it's unpinned privilege-boundary code. Med.
7. **/health is unthrottled and runs deep checks** (DB ping + S3 HeadBucket per unauthenticated request) — cheap amplification vector; fixed as part of top-5 item 4.

Adjacent: the missing backups (top-5 item 2) are a data-loss/integrity exposure, and the unverified rbac magic-rank-literals item would become security-relevant the first time a role is inserted between existing ranks.
